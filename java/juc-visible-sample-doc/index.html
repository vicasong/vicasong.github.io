Java共享变量可见性的一些简述

最近有同事在线上遇到多线程共享变量可见性问题，简单处理后，我觉得这些基础应该需要再整理总结一下。（该部分内容需要熟悉JVM并发相关概念和了解JVM解释运行过程，有一定阅读门槛）

由于Java内存模型（JMM）定义了Java程序中变量会存储在主内存中，而每个线程都有自己独立的工作内存，其中保存了当前线程使用到的变量副本。
即，不同线程之间是无法访问其他线程的工作内存中的变量的，共享变量在线程之间的传递需要通过主内存完成。同时，因为CUP有很多级缓存（L1 Cache、L2 Cache等），而线程可能运行在不同的CPU核心上，
在一个特定时间它们读取到的“内存”内容可能会不一样。

由此就会引出一个问题，多线程对共享变量修改导致不一致。例如：

设x=1，线程0、线程1分别计算x=x+1，期望会得到x=3，但实际运行如下：

1 thread-0 读取 x=1 x=1
2 thread-0 计算 x=2 x=1
3 thread-1 读取 x=1 x=1
4 thread-0 写回 x=2 x=2
… … .. … …

那么有聪明的小伙伴就说了，我等线程0操作完再继续线程1的计算不就好了嘛：

但是，此时有一个优化策略就出来搞事了：“指令重排”，即编译器或CPU为了提高程序效率会将部分指令的执行顺序进行调整，也就是指令不一定会按照书写顺序执行：

… … .. … …

那么我们如何才能保证共享变量的可见性呢？

可见性

在现代CPU及硬件架构中，缓存一致性协议会保证不同CPU操作的数据缓存始终是一致的。

缓存一致性

CPU缓存行

速度越快的存储，其硬件成本越高。为了权衡成本效益，存储会被分成多层架构：寄存器、一级缓存（L1 Cache）、二级缓存（L2 Cache）、三级缓存（L3 Cache）、内存和硬盘等，其速度、容量和成本依次下降。

缓存一致性协议

当多个cpu对同一块内存地址进行操作时，即共享数据时，如何保证它们操作的数据的一致性呢？

缓存一致性，即当CPU0对缓存行中的数据做了修改，就要通知其他CPU，例如CPU1，它的缓存行失效了，那么CPU1想要操作该缓存行的数据时就会退行到从下一级存储重新加载数据。

缓存一致性协议有很多种不同的实现，但此处主要简述一下典型的MESI协议场景。

在MESI协议中，每个缓存行都会有一个固定状态：Modified、Exclusive、Shared、Invalid。
假设单前有2个CPU核心：A和B，当A从存储中加载一个变量时，会读取一段连续的数据，即缓存行，该缓存行将会被标记为“Exclusive”，此时只有A在操作该缓存行（唯一的数据缓存拷贝，且数据和主内存是一致的），数据会持续保留在缓存中（对缓存的修改不会立即刷新到下级存储，仅在有必要的时候）。
此时，B也要读取一个变量，刚好和A读取的数据在同一块缓存行中，那么此时B的缓存中也缓存了同样的缓存行，但A、B缓存行都会被标记为“Shared”（缓存数据和主内存依旧是保持一致的）。
然后我们假设A接下来对它读取的变量进行了修改，那么A的缓存行会被标记为“Modified”（此时，该缓存行的数据就和主内存不一致，且仅该缓存行的数据才是最新的），同时A会通知B说你的缓存行过时了，此时B的缓存行会被标记为“Invalid”（此缓存行已过期，缓存的内容不会再被使用）。
那么过了一段时间，B要再次读取它的变量了，但B此时的缓存行已经失效了，它应该去下一级存储重新读取数据，B对该数据块的读取访问会强制要求A将“Modified”标记的该缓存行刷新到下一级存储，等数据刷新后B读取到的就是最新的缓存行，此时A、B的缓存行再次被标记为“Shared”。

伪共享（False Sharing）

而如果它们修改的是缓存行中不同区域的数据时，就会争用同一个缓存行，导致看起来像是争用同一个共享变量一样。

缓存行会因此被不停标记为失效，并刷新和重新加载，这会带来性能问题。因此，对于同一块数据块上的不同变量的操作，可以通过填充数据使它们分布到不同的缓存行上来避免此性能问题。

变量易失性

我们有2个线程会分别执行上面的对应方法，如代码所写：线程1将无限循环直到线程0将共享变量ready的值做出修改，从而终止线程1的循环。

然而事情并没有这么简单，首先每个线程都会有共享变量的副本（JMM）保存在各自的堆栈中，JMM是一个抽象概念，这些数据可能在物理内存中，也可能在CPU缓存或者寄存器中。
而当线程0对共享变量进行修改后，线程1并不能看到该变量的修改，它仍将保持无限循环。

这里就有人开始困惑了，前面不是说了在硬件层面，CPU缓存会保持一致么，比如：MESI协议。那么线程共享的数据为什么还会不一致呢？
问题就在这里，如上所述“缓存一致性”，即是关于CPU缓存的，数据被层层加载到CPU中，每个CPU都有自己的寄存器，缓存会通过MESI等协议保持一致，但寄存器例外。
编译器很“聪明”，它认为在单线程下，变量ready不会被修改，此变量会存储在CPU寄存器中，因为寄存器的访问速度要比缓存快的多的多。但现在是多线程环境，编译器也会假设即使是多线程，也不会有数据通信，
每个线程都会复制一份数据的副本并存储在寄存器中，如果一个线程访问数据，这份数据刚好存在于寄存器中，那么在其他CPU的线程就无法看到数据的更新。

所以，有办法阻止编译器将共享变量存储到寄存器中吗？这就是volatile（具体是如何做的，请继续往下看）.

网络上有很多关于易失性的文章，都提到了volatile会强制让数据交换使用主内存，这是过时或错误的描述，且一直被广泛传播。

  1. 首先，在Java1.5时代，主流CPU架构中，缓存一致性是可选功能且默认不开启，volatile为此担任了一个重要角色就是使用指令开启缓存一致性。但如今该功能已成为架构中的必选项，缓存一致性是硬件层协议，软件开发者无需额外处理。
  2. volatile不会也无法直接让数据从主内存中读写，因为效率太低了，CPU要等待数据从主内存读取或写入就好像你要从澳洲步行和游泳到俄国北部，和那里的朋友说一句话然后再徒步返回，慢到令人发指。因为如今CPU频率都是至少GHz为单位，而主内存频率只是Mhz，它们速度差距几乎是跨维度的。
  3. CPU存储架构对其上的应用是透明的，JMM足够抽象，其中的数据可以在任何位置（寄存器、缓存或主内存等等），volatile不能使CPU立即刷新缓存，系统会在必要的时候进行缓存刷新操作，由缓存一致性保证数据在缓存中是始终一致的。

顺序性

依旧是volatile，该关键字标记的变量会在编译器或CPU进行指令优化时做出一些限制。涉及到几个屏障（barriers）：LoadStore、LoadLoad、StoreLoad、StoreStore。

缓存一致性&内存一致性

缓存一致性协议确保对同一内存位置的读取将始终返回最新的值。仅当存这个同一内存位置的缓存有多个副本时，才需要缓存一致性协议，其工作内容就是保持所有副本的一致性，以下注意区分缓存和缓冲。

内存一致性模型则关注于读取和写入到不同内存位置的相对顺序，TSO（Total Storage Order）。

顺序

由于X86架构CPU提供了TSO，所以LoadStore、LoadLoad、StoreStore这几个屏障相当于空操作，主要用于限制编译器优化，StoreLoad会有额外指令实现。

屏障类型 示例指令 说明
LoadLoad Load1;LoadLoad;Load2 该屏障确保Load1数据的加载先于Load2及其后所有读取指令的操作
StoreStore Store1;StoreStore;Store2 该屏障确保Store1的写入操作先于Store2及其后所有写入指令的操作
LoadStore Load1;LoadStore;Store2 确保Load1的数据加载先于Store2及其后所有的写入指令的操作
StoreLoad Store1;StoreLoad;Load2 确保Store1的写入操作先于Load2及其后所有读取指令的操作。同时该屏障之前的所有数据访问指令完成之后，才执行该屏障之后的数据访问指令

X86是一种处理器架构，我们用的大多数CPU都是X86架构，Intel和AMD生产的绝大多数CPU，比如奔腾（Pentium），赛扬(Celeron)，酷睿(core)，Xeon等等都是该架构，而X86_64是X86的扩充，并且兼容X86

补充

在有些平台的JVM实现中，long、double等64位数据的读写会被拆成两次32位读写操作，在多线程并发下可能会导致读取或写入不一致的问题，使用volatile可以保证该类型变量读写原子性。但实际上是有性能损耗的（volatile还保证了顺序性），即使在X86架构64位机器上，其已经保证了对64bit数据的读取和写入是原子性的，但我们无法保证程序不会在32位机器上运行，所以对long、double使用volatile是有意义的。
